01Purpose, Scope & Normative References

1.1 Purpose

This guideline establishes a repeatable method for translating software requirements into Module/Unit Design (MUD) flowcharts within AUTOSAR Classic Platform projects. It targets the SWE.3 — Software Detailed Design and Unit Construction process area defined by Automotive SPICE, ensuring that every flowchart produced is reviewable, traceable to its source requirements, and compliant with ISO 26262 Part 6 safety expectations.

1.2 Scope

This document applies to the detailed design of application-layer SWC Runnables and Complex Device Driver (CDD) units in AUTOSAR-based ECU software. It covers functional logic flow, decision structures, error handling paths, and timing-related patterns. It does not cover BSW module configuration or system-level architecture diagrams (which belong to SWE.2).

A-SPICE SWE.3 Context
The detailed design flowchart is a key work product for SWE.3-BP.1 (develop software detailed design) and SWE.3-BP.3 (describe dynamic behavior). It must be created before unit implementation and serve as the specification against which SWE.4 unit tests are written.

04Requirement-to-Flowchart Conversion Process

The following seven-step process transforms software requirements into a complete, standards-compliant MUD flowchart. Each step maps directly to A-SPICE SWE.3 base practices and ISO 26262 Part 6 design method expectations.

4.1 The Seven-Step Conversion Method

1

Identify Inputs, Preconditions & Trigger Events

Extract all "WHEN", "IF", "UPON", and "EACH" clauses from requirements. These define the entry conditions, triggering RTE events (TimingEvent, DataReceivedEvent), and the input data flowing through Rte_Read operations. Map each to an Input/Output parallelogram at the top of your flow.

SWR_SPD_001 "The module shall read the vehicle speed signal from the RTE each 10 ms task cycle." → Block: I/O Parallelogram — "Read VehicleSpeed via Rte_Read" → Trigger: TimingEvent (10 ms)
2

Classify Requirements into Behavior Categories

Sort every requirement into one of four categories. This classification drives the flowchart structure and ensures completeness of coverage:

CategoryDescriptionFlowchart Region
A. NominalNormal operating logic — the happy pathCenter/main flow
B. AlternativeSpecial conditions, mode-dependent pathsBranching from decisions
C. Error / SafetyFault detection, DEM reporting, degradationRight-side error branches
D. Initialization / StateInit sequences, mode transitions, debouncingTop section or separate sub-flow
3

Establish the High-Level Flow Skeleton

Begin every Runnable flowchart with this canonical structure — it matches automotive design conventions and audit expectations across OEMs:

Figure 2 — Canonical Runnable Flowchart Skeleton
4

Map Each Requirement to a Block Type

Apply this conversion table to every individual software requirement:

Requirement PatternBlock TypeExample
"…shall compute / calculate / filter…"Process (rectangle)"Compute torque request"
"…if / when / in case of…"Decision (diamond)"Speed > MaxSpeed?"
"…shall read / receive…"Input (parallelogram)"Read ADC channel 3"
"…shall output / write / send…"Output (parallelogram)"Write PWM duty cycle"
"…shall call / invoke…"Predefined Process"Call CheckSensorValidity()"
"…shall report / set DTC…"Process + Annotation"Report DEM event" + [SWR_ERR_012]
Golden Rule
Every requirement shall map to at least one flowchart element. If a requirement does not appear in the flowchart, either the flowchart is incomplete or the requirement belongs to a different design level.
5

Connect Blocks with Directed Flow Lines

Apply these layout rules strictly: primary flow runs top-to-bottom; sub-flows branch left-to-right; error paths branch rightward and are drawn with dashed lines or distinct color. Every flow line must have an arrowhead. Every decision exit must carry a label (Yes/No, True/False, or the specific condition name).

6

Add Error & Safety Handling Paths

ISO 26262 Part 6 requires explicit documentation of error detection, degradation, and safe-state transitions. For every decision where a fault or invalid condition is possible, add a clearly labeled error branch showing:

a) The specific error flag or DEM event to set, b) The safe/default output value, c) The termination point (END or transition to safe state).

7

Annotate with Requirement IDs & ASIL Tags

Attach requirement ID annotations (using bracket symbols) alongside every flowchart element they trace to. For safety-relevant paths, add ASIL level tags. This enables direct traceability from flowchart to requirements (SWE.3-BP.5) and supports A-SPICE assessment evidence.

05AUTOSAR-Specific Design Patterns

5.1 Runnable Entry Pattern

Every AUTOSAR Runnable is triggered by an RTE event. The flowchart for a cyclic Runnable should begin with the triggering context and immediate port reads:

START: Rnbl_SpeedCtrl_10ms ├── Read VehicleSpeed ← Rte_Read_RPort_VehicleSpeed_VehSpd() ├── Read TargetSpeed ← Rte_Read_RPort_TargetSpeed_TgtSpd() ├── Read EnableFlag ← Rte_Read_RPort_CtrlEnable_EnblFlg() │ ... [processing logic] ... ├── Write TorqueCmd ← Rte_Write_PPort_TorqueCmd_TrqReq() └── END

5.2 Common AUTOSAR Flow Patterns

Pattern B — Mode-Dependent Processing (Init / Normal / Fault)

When a module operates in distinct modes (common in motor control, HVAC, and BMS applications), use a top-level multi-way decision that dispatches to separate sub-flows. Each mode path should be self-contained with its own output write and state update.

Pattern C — Cascaded Safety Checks (ASIL Decomposition)

For ASIL-B and above, safety checks must be decomposed into individual decision diamonds rather than combined into a single complex condition. This supports structural coverage analysis (MC/DC) required by ISO 26262 Part 6 Table 9.

5.3 AUTOSAR Service Call Integration

When a Runnable interacts with BSW services (DEM, NvM, FIM, WdgM), represent these as Predefined Process blocks (rectangle with double vertical edges) and label them with the standardized AUTOSAR API name:

06Mandatory Design Rules (Automotive Grade)

The following rules are non-negotiable for any MUD flowchart intended for A-SPICE assessment or ISO 26262 certification evidence. Violations will be flagged during design reviews and formal assessments.

6.1 Structural Integrity Rules

Rule R-01 — Complete Exits

Every decision diamond shall have all possible outcomes explicitly labeled. Boolean decisions require exactly two exits (Yes/No). Multi-way decisions require one exit per enumeration value plus a default/else path.

Violation Example

A diamond with only one arrow leaving it, or exits labeled only "Yes" with no "No" path visible. This creates ambiguous behavior and will fail A-SPICE BP.4 evaluation.

Rule R-02 — No Dangling Lines

Every flow arrow must terminate at a valid block (process, decision, I/O, connector, or end). No arrow shall point into empty space or stop without connection.

Violation Example

An error branch arrow that points downward but does not connect to any block — the reviewer cannot determine what happens after the fault is detected.

Rule R-03 — No Hidden Logic

The flowchart shall express the complete algorithmic behavior of the unit. Any logic present in the code but absent from the flowchart constitutes a design gap.

Violation Example

A process block labeled "Apply filter" that omits the type of filter (FIR, moving average, low-pass) and its relevant parameters. The design is not detailed enough for independent unit test derivation.

Rule R-04 — Decompose Complex Decisions

Conditions with more than two Boolean sub-expressions shall be decomposed into sequential diamonds. This supports MC/DC coverage analysis.

Violation Example

A single diamond: "InputValid AND !Timeout AND !FaultActive AND Speed < 200?" — must be broken into four sequential decisions.

6.2 Naming & Labeling Rules

Rule R-05 — Action Verb Labels

Process blocks shall begin with an action verb: Compute, Filter, Update, Validate, Check, Limit, Increment, Reset, Set, Clear, Call, Report.

Violation Example

"Torque value" — this is a noun phrase, not an action. It gives no indication of what the unit does with the torque value.

Rule R-06 — Design Level Abstraction

Flowcharts shall operate at the design level, not implementation level. Use physical/functional names (e.g., "Motor Speed"), not C variable names (e.g., u16_MtrSpd_rpm).

Violation Example

"if (u16_MtrSpd_rpm > CAL_MaxSpd_rpm)" — this is code, not design. The flowchart should read: "Motor Speed > Maximum Speed Limit?"

Rule R-07 — One Sentence Per Block

Each process block shall contain a single, concise statement (maximum one sentence). If more detail is needed, decompose into multiple blocks or reference a sub-flow.

Rule R-08 — Requirement Traceability Tags

Every flowchart element derived from a requirement shall carry an annotation with its source requirement ID (e.g., [SWR_MOD_042]). This is mandatory for A-SPICE SWE.3-BP.5 bidirectional traceability.

6.3 Layout Rules

Rule R-09 — Top-to-Bottom Primary Flow

The nominal execution path shall flow from top to bottom. Error/alternative branches extend to the right. Loops return flow upward using connectors (circles), never by drawing arrows that cross the main flow.

Rule R-10 — Hierarchical Decomposition

If a single flowchart exceeds one printed page (approximately 15–20 blocks), decompose it into a Level 0 overview calling Level 1 sub-flows via Predefined Process blocks.

07Worked Example — Speed-Torque Control Module

7.1 Source Requirements

SWR_TRQ_001 The Runnable Rnbl_TrqCtrl_10ms shall read the vehicle speed signal each 10 ms cycle. SWR_TRQ_002 The module shall validate the speed input. Valid range: 0 to 250 km/h. SWR_TRQ_003 If the speed input is invalid, the module shall output a torque of 0 Nm and report DEM event DTC_SpdInvalid. SWR_TRQ_004 If the speed input is valid, the module shall compute the torque request using the control algorithm. SWR_TRQ_005 The computed torque shall be limited to the range ±300 Nm. SWR_TRQ_006 The module shall write the final torque command to the RTE output port. SWR_TRQ_007 The module shall store the current speed value for use in the next cycle.

Common Assessment Finding
A-SPICE assessors frequently downgrade SWE.3 capability when the detailed design is derived from code (bottom-up) rather than from requirements (top-down). The flowchart must be created before implementation. Evidence of this includes timestamped version control commits showing the design document predating the corresponding source code.

9.2 PDF Export Tips

When exporting flowcharts for formal documentation or assessment evidence, ensure: vector format output (SVG or PDF, not rasterized PNG/JPEG) for readability at any zoom level; consistent page size (A3 landscape is recommended for complex flows); legible font size (minimum 8pt for annotations, 10pt for block labels); and a title block with document ID, version, author, date, and ASIL classification.
